Как работает авторизация VK ID на Android
Мы предлагаем следующие способы авторизации:
через SDK c обменом кода на фронтенде;
через SDK с обменом кода на бэкенде;
без SDK.

Через SDK с обменом кода на фронтенде

Пользователь инициирует процесс авторизации: нажимает кнопку Войти через VK ID.
VK ID SDK генерирует PKCE с набором параметров, необходимых для защиты от CSRF-аттак:
codeVerifier — случайно сгенерированная строка, новая на каждый запрос авторизации. Может состоять из следующих символов: a-z, A-Z, 0-9, \_, -. Длина от 43 до 128 символов;
codeChallenge — codeVerifier, зашифрованный с помощью метода S256 и закодированный в base64.
Также генерируется параметр state — произвольная строка состояния приложения.
VK ID SDK открывает WebView https://id.vk.ru/authorize, используя несколько параметров. Основные из них:
response_type=code — требуемый ответ. При запросе кода подтверждения следует указать значение code;
client_id — ID приложения, полученный при регистрации;
redirect_uri= vk{clientId}://vk.ru/blank.html — адрес для возврата в приложение Партнёра.
Также используются параметры codeChallenge и codeVerifier, созданные на шаге 2.
Пользователю отображается форма авторизации в WebView.
Пользователь проходит аутентификацию.
Запрос пользователя на авторизацию в сервисе Партнёра передается из VK ID Frontend (далее — WebView) в VK ID Backend.
VK ID Backend аутентифицирует пользователя и генерирует код авторизации.
VK ID Backend передаёт через Redirect URI в WebView код авторизации и параметр state, созданный на шаге 2.
VK ID SDK получает через WebView код авторизации и state через Redirect URI.
VK ID SDK проверяет полученный state.
VK ID SDK запрашивает обмен кода авторизации на Access token. Для этого передается:
код авторизации;
codeVerifier из PKCE;
Redirect URI.
VK ID SDK получает Access token, привязывает его к IP-адресу Пользователя и передаёт Партнеру.
Партнер получает Access token — авторизация успешно завершается.
Если IP-адрес Пользователя изменился, токен станет невалидным и его нужно обновить. Для этого воспользуйтесь методом vkid.refreshToken(). Подробнее.

Обратите внимание
Генерировать параметры PKCE нужно на фронтенде, а передавать на бэкенд — только при обмене кода на токены.
Пользователь инициирует процесс авторизации: нажимает кнопку Войти с VK ID.
Клиентское приложение генерирует PKCE с набором параметров, необходимых для защиты от CSRF-аттак:
codeVerifier — случайно сгенерированная строка;
codeChallenge — codeVerifier, зашифрованный с помощью метода S256 и закодированный в base64.
Также генерируется параметр state — произвольная строка состояния приложения.
Подробнее о передаче параметров PKCE.
Клиентское приложение инициализирует безопасную авторизацию с использованием параметров PKCE — отправляет запрос в VK ID SDK.
Запрос клиентского приложения передается из VK ID SDK в VK ID Frontend (далее — WebView).
WebView перенаправляет на id.vk.ru/authorize.
VK ID Backend запускает форму авторизации на WebView.
WebView показывает пользователю форму авторизации.
Пользователь проходит авторизацию.
Запрос пользователя на авторизацию в сервисе Партнёра передается из WebView в VK ID Backend.
VK ID Backend аутентифицирует пользователя и передаёт через Redirect URI в WebView код авторизации code, уникальный идентификатор вашего мобильного устройства deviceId и параметр state, полученный на шаге 3.
WebView передаёт через Redirect URI в VK ID SDK код авторизации, deviceId и параметр state.
В callback onAuthCode приходит параметр data, в котором находятся код авторизации, deviceId и параметр state.
Клиентское приложение передаёт в Backend клиентского сервиса код авторизации, codeVerifier, deviceId и параметр state.
Backend клиентского сервиса запрашивает у VK ID Backend замену кода авторизации на Access Token V2, Refresh Token V2 (далее — ATv2 и RTv2) и ID Token.
Backend клиентского сервиса получает от VK ID Backend ATv2, RTv2 и ID Token в обмен на код авторизации — авторизация успешно завершается. Также происходит привязка ATv2 к IP-адресу Backend клиентского сервиса.
